Deferred hardening for a future v2.1. V2 shipped as `v2.0.0` at commit
`1218cd7` on 2026-07-23; every release blocker below is closed and kept only
as historical context. Do not work the deferred items unless the user
explicitly brings them back into scope.

support. It is not required to be a research-grade distributed experimentation
platform.

## V2 release blockers

### R1. Restore a green repository gate

Source: current filesystem-store work.

Fix the remaining lint failure. V2 is ready only when tests, typecheck, lint,
and the generated-artifact style gate all pass.

### R2. Judge only closed source artifacts

Source: concern 47.

The judge validates event shapes, run identity, and sequence numbers, but
`validateCanonicalSequence` does not require terminal closure. Removing the
final `run.completed` event from a valid run still lets the judge return a known
score.

For v2, require the final source event to be exactly one `run.completed` or
`run.failed` event before calling the judge. Add the focused nonterminal-prefix
test.

Record persistence should also survive an `agent.dispose()` failure. Persist
the completed or failed evaluation record, then report the cleanup error.

### R3. Never turn unknown paid usage into zero spend

Sources: concerns 40 and 49.

`partialSpend` catches a pricing error for a nonterminal failed run and returns
zero. Repeated infrastructure failures can therefore release their
reservations without advancing aggregate spend.

The reliability collector applies fail-closed unknown-cost handling only when
its caller supplied `maxTotalAmount`. It also adds `reasoningTokens` on top of
`outputTokens`, although reasoning is a subset under the normal domain budget
rule. Its pre-dispatch check can let the next sample cross a declared ceiling.

For v2:

- Under fail-closed accounting, unknown partial spend must halt further paid
  dispatch or conservatively charge the reserved amount.
- Reliability collection must reject unknown price regardless of whether a
  monetary ceiling argument was supplied.
- Token accounting must match the domain rule and not count reasoning twice.
- Do not dispatch the next reliability sample unless a conservative bound for
  it fits the remaining declared budget.

Keep these fixes narrow. Per-attempt routing identity and research provenance
are deferred.

### R4. Reward output must be finite and must not invent measurements

Source: concern 50.

`monetary.maxAmount: NaN` currently passes the `<= 0` check and produces a
known reward whose monetary fraction and scalar are `NaN`.

The engine also supplies zero when positively weighted token or latency
measurements are unavailable:

```ts
tokensUsedFraction: usageScore?.status === "known" ? 1 - usageScore.score : 0,
latencyFraction: 0,
```

For v2:

- Require `monetary.maxAmount` to be finite and positive.
- Use the recorded latency evaluator result instead of a fixed zero.
- If a positively weighted token or latency measurement is unavailable, return
  an unavailable reward rather than substituting zero.
- Add a final finite-number assertion for every known measurement, term, and
  scalar emitted by the engine.

The existing completion score may remain the v2 quality measurement. A full
rubric quality vector is deferred.

## Explicitly deferred to v2.1

These are valid hardening ideas, but they do not block the v2 product boundary:

- Concern 34: execution-time Git verification, attestation persistence in every
  artifact, and complete reliability preregistration.
- Concern 35: a fully general fixed study configuration, every participant
  capability combination, and exhaustive matrix preflight.
- Concern 40: exact effective model identity for each retry attempt.
- Concern 41: distributed stale-lease takeover, owner-token release, directory
  sync guarantees, and exhaustive path-component hardening. Document the v2
  filesystem store as single-worker rather than claiming distributed safety.
- Concern 47: expanded judge policy and request provenance beyond the exact
  prompt, controls, response, usage, and source identity already recorded.
- Concern 48: paired and stratified reliability statistics, full sampling
  design, blinded labels, all judge populations, raw rubric vectors, and exact
  research-artifact matching.
- Concern 49: research-grade prompt and sampling provenance beyond the narrow
  spending fixes in R3.
- Concern 50: dimension-level reward vectors with units, group scope,
  normalization identities, and a separate vector-construction pipeline.

Reliability studies and automated optimization should be labeled experimental
in v2. They must remain opt-in and must not be used to make a strong comparative
research claim.

## Accepted v2 foundation

The following work is sufficient foundation for the release:

- Versioned tool policies are recorded and enforced through the dispatcher.
- Canonical events, tool traces, replay checks, and artifact identity checks are
  implemented.
- Study execution has bounded concurrency, resume, stopping rules, terminal
  failure retention, and aggregate reservations.
- Debate and judge options are snapshotted before asynchronous execution.
- Requested controls and evaluation records use exact validation.
- The evaluator port is shared by deterministic and judge-backed evaluators.
- Reliability analysis is opt-in, preserves missing evaluations, and no longer
  converts absent comparison populations into measured zero effects.
- Reward scalarizer identity is resolved from the study declaration and
  retained under a full digest.

Once R1-R4 pass, stop implementation expansion and prepare the v2 release.
